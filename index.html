Corporate Finance Transformation

Nov 2023 – May 2025
BMO · enterprise data migration

Moved 183 on-prem legacy schemas to AWS with full historical and real-time accuracy — replacing hand-written pipelines with one metadata-driven framework.

75%less manual coding
90%reliability gain
183schemas migrated
GlueLambdaRedshiftAirflowCDK
Full breakdown
  • The problem: finance data lived in 183 separate on-prem legacy schemas. Every new schema needed its own hand-coded pipeline — slow to build, inconsistent, and painful to audit.
  • What I built: a metadata-driven ingestion framework on Glue, Lambda, Redshift, S3 and Airflow. Instead of writing a new pipeline per schema, the framework reads metadata definitions and generates Airflow DAGs dynamically — one codebase, 183 schemas, both historical backfills and real-time loads.
  • Data quality: implemented automated quality checks and alerting in SQL and AWS services — row counts, schema drift, freshness — catching issues before downstream consumers ever saw them, lifting reliability by 90%.
  • Delivery: led CI/CD through GitHub Actions and AWS CDK, so every release was reproducible, reviewed, and audit-ready — critical in a banking environment.
  • The result: manual coding effort dropped 75%, and onboarding a new schema went from a development project to a config change.

Enterprise Data Lake Modernization

Feb 2025 – Present
BMO · legacy decommissioning

Migrating L1 and L3 schema-level tables from Hadoop/Netezza to Redshift, retiring legacy systems and accelerating analytics for downstream teams.

60%faster deployments
2legacy systems retired
PySparkAirflowJenkinsDockerCloudFormation
Full breakdown
  • The problem: aging Hadoop and Netezza platforms were expensive to run, slow for analysts, and increasingly unsupported. The L1 (raw) and L3 (curated) layers had to move to Redshift without breaking anything downstream.
  • What I built: PySpark transformation jobs orchestrated by Airflow DAGs driven entirely by JSON config files — schema mapping, type conversion and data validation are declared in config, not buried in code, so the same engine migrates table after table.
  • Validation: every migrated table is reconciled against its source — row counts, checksums and business-rule checks — so legacy systems could be decommissioned with confidence.
  • Delivery: built CI/CD pipelines with Jenkins, Docker and AWS CloudFormation, cutting deployment time by 60% and making infrastructure changes reviewable and repeatable.
  • Architecture: worked directly with data architects to keep migration patterns scalable and cloud costs in check as table volume grew.

Full breakdown
